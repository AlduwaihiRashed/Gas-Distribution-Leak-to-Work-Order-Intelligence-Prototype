# Just To Be On Track

**1. What we did already**
- Step 1: made fake but realistic pipe sensor data (pressure, gas %, sound), cleaned it up, and wrote code that watches it and flags anything that looks like a leak. One weird sensor alone gets a smaller alert, two weird sensors together get a bigger alert. Caught every real leak with zero false alarms.
- Step 2: built a simple rulebook that turns each flagged leak into Grade 1 (emergency), 2 (scheduled repair), or 3 (monitor) based on gas amount, location, and ground cover (paved surfaces trap gas and force Grade 1 even at low readings). 100% match against our test answers.
- Step 3: built the "lookup the rulebook" step. We chunked our safety/procedure documents, turned them into searchable vectors (ChromaDB), and now every graded leak automatically pulls the real procedure text that justifies its grade — so the work order can cite something real instead of just stating a grade with no explanation. Along the way we found our source documents didn't actually have real leak-response procedures in them (just architecture diagrams), so we wrote a proper procedures reference doc and added it to the searchable corpus.
- Step 4 (done): re-ran the model bakeoff from scratch against the real 40-second budget (the old ministral-3 pick assumed ~2min/work order was fine, which it no longer is). Tested all 8 locally-installed models — llama3.2 (1b/3b), qwen3.5 (0.8b/2b), gemma4 (e2b/e4b), deepseek-r1 (1.5b/8b) — two incidents each (a Grade 1 and a Grade 3), on the actual synthesis prompt with JSON-schema-constrained output. Results: qwen3.5 doesn't reliably honor the schema constraint (produced invalid JSON both times, both sizes); deepseek-r1 burns its whole token budget on hidden reasoning even with thinking disabled, so it never got to write the work order (both sizes); llama3.2:1b is fast (~3s) but hallucinated its citations instead of citing the real retrieved text, which fails the "must cite something real" rule. Settled on **gemma4:e2b** as the one model to run with — best groundedness (richest, most specific use of the retrieved procedure text) and still comfortably inside budget. Also tightened the JSON schema so `citations` can only be one of that incident's *actual* retrieved chunk ids (closed a near-miss where a model cited the right doc but tacked on a hallucinated `#anchor`). Final run over all 6 gradable incidents: 100% valid JSON, 100% correctly-cited, latency 5.3s–8.4s (avg 7.3s) — nowhere near the 40s ceiling. Wrote `src/synthesize_work_order.py` (the real M7 pipeline step — model is still a `--model` flag, so the "configurable accuracy variable" the blueprint wants is there if we ever need to swap; also writes a `synthesis_evaluation.json` alongside the work orders, same pattern as the other stages) with the degraded-template fallback for when Ollama is down (tested by pointing it at a nonexistent model, confirmed it still emits a safety-complete, banner-flagged work order in ~2ms), and `src/bakeoff_synthesis_models.py` (the reusable bakeoff script, kept around in case we add/swap models later). Full bakeoff evidence in `data/synthesis_output/model_bakeoff.json`; final outputs in `data/synthesis_output/work_orders.json` + `synthesis_evaluation.json`.

- Step 5 (done): n8n was set up locally via podman (`podman run ... docker.io/n8nio/n8n`, data in a persistent `n8n_data` volume). Since the pipeline stages were CLI scripts writing JSON files, there was nothing for n8n to call — wrote `src/api.py` (M8), a small FastAPI service that fronts the pipeline: `GET /incidents/pending` (graded incidents not yet queued), `POST /work-orders/synthesize` (the one auto-create-and-auto-route action, calls M7 and drops the result straight into `data/work_order_queue/queue.json` — no separate "create" step exists, since splitting it would reintroduce a manual trigger R5 rules out), and `POST /work-orders/{id}/approve` (the *only* human gate — authorizes physical dispatch, never creation/routing). Built two n8n workflows against it via n8n's REST API (had to generate an API key from the n8n UI for this): **"L2WO - Auto Work Order Routing"** (schedule trigger polls every 10s, no manual trigger needed to start it) and **"L2WO - Dispatch Approval Gate"** (webhook a future dashboard "Approve" button will call). Saved both under `n8n/` for reproducibility.
  Testing this for real (not just assuming the JSON would import and work) surfaced three real bugs, all fixed: (1) a race where n8n's poll could overlap with a still-in-flight synthesis call for the same incident, causing duplicate Ollama calls and pushing one incident's latency to 42.8s — fixed by having the API claim an incident (`IN_PROGRESS` status) *before* the slow LLM call, not after; (2) the fail-safe branch was only wired onto the synthesis HTTP node, so a *total* API outage (not just an Ollama failure) crashed the whole n8n execution instead of routing to a fallback — added a second fail-safe branch off the earlier `/incidents/pending` call, tested by killing the API outright and confirming n8n logs a banner alert instead of just failing silently; (3) n8n's file-write node has a genuine bug where "append" opens the file with an invalid flag combination that always fails, and separately its default `N8N_RESTRICT_FILE_ACCESS_TO` only allows writes under `~/.n8n-files` — worked around both (write-with-overwrite instead of append; correct target directory). Confirmed the full loop end-to-end: all 6 incidents auto-routed with zero duplicate Ollama calls, human approval verified through the actual n8n webhook (double-approve correctly 409s back through n8n's error branch), and after killing + restarting the API mid-run, the pipeline self-healed with no manual intervention. One honest, undecided limitation to flag in the report: since Ollama serializes requests on this single GPU, a *burst* of several incidents graded at the same instant will queue internally and the last one in the burst can approach/exceed the 40s budget — not a bug, just the real cost of "one local GPU, one model instance," worth mentioning in the latency-SLA writeup (Part D).

Also improved (not a bug, but worth noting): the synthesis prompt now gets told the actuator's confirmed state when available, so the work order says "already isolated" instead of generic "dispatch crew to isolate" text that would've been redundant, if not contradictory, for an already-actuated segment.
